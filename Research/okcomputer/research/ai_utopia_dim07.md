### 7.1 Voyager: Skill Library + Automatic Curriculum

**Claim: Voyager consists of three key components: an automatic curriculum that maximizes exploration via in-context novelty search, an ever-growing skill library of executable code, and an iterative prompting mechanism with self-verification.** [^1^]
**Source:** Voyager: An Open-Ended Embodied Agent with Large Language Models (Wang et al., NeurIPS 2024 / ICLR 2025 Journal Track)
**URL:** https://arxiv.org/abs/2305.16291
**Date:** 2023-05-25
**Excerpt:** "Voyager consists of three key components: 1) an automatic curriculum that maximizes exploration, 2) an ever-growing skill library of executable code for storing and retrieving complex behaviors, and 3) a new iterative prompting mechanism that incorporates environment feedback, execution errors, and self-verification for program improvement."
**Context:** Voyager uses GPT-4 as a blackbox LLM queried via prompting. It bypasses any parameter fine-tuning.
**Confidence:** high

**Claim: Voyager's automatic curriculum generates tasks based on the overarching goal of 'discovering as many diverse things as possible,' taking into account the agent's current state (inventory, equipment, nearby blocks, biome, time, health) and exploration progress (completed/failed tasks).** [^1^]
**Source:** Voyager (Wang et al., 2023)
**URL:** https://voyager.minedojo.org
**Date:** 2023
**Excerpt:** "The curriculum unfolds in a bottom-up fashion, allowing for considerable adaptability and responsiveness to the exploration progress and the agent's current state... The curriculum is generated by GPT-4 based on the overarching goal of 'discovering as many diverse things as possible'. This approach can be perceived as an in-context form of novelty search."
**Context:** The curriculum is environment-aware, querying game state before proposing next task.
**Confidence:** high

**Claim: Voyager's skill library stores each skill as executable JavaScript code indexed by the embedding of its natural-language description. Retrieval fetches top-5 most relevant skills by embedding similarity.** [^1^]
**Source:** Voyager (Wang et al., 2023)
**URL:** https://arxiv.org/html/2305.16291
**Date:** 2023
**Excerpt:** "Each program is indexed by the embedding of its description, which can be retrieved in similar situations in the future... When faced with a new task proposed by the automatic curriculum, we perform querying to identify the top-5 most relevant skills."
**Context:** Skills are stored as code, not natural language descriptions. Execution is deterministic code.
**Confidence:** high

**Claim: Removing the automatic curriculum dropped discovered-item count by 93%. Removing self-verification dropped performance by 73%. The skill library matters most in later stages (80+ iterations), where agents without it plateau.** [^2^]
**Source:** Voyager ablation studies (Wang et al., 2023)
**URL:** https://beancount.io/bean-labs/research-logs/2026/05/08/voyager-open-ended-embodied-agent-lifelong-learning
**Date:** 2023
**Excerpt:** "Removing the automatic curriculum dropped discovered-item count by 93%. Removing self-verification dropped performance by 73%. The skill library matters most in later stages -- early on it helps little; at 80+ iterations, agents without it plateau."
**Context:** Critical ablation results showing curriculum is essential for consistent progress.
**Confidence:** high

**Claim: Voyager discovered 63 unique items across 160 prompting iterations, 3.3x more than prior SOTA. It was the only method to reach diamond-tier at all.** [^2^]
**Source:** Voyager (Wang et al., 2023)
**URL:** https://arxiv.org/abs/2305.16291
**Date:** 2023
**Excerpt:** "Voyager discovered 63 unique items across 160 prompting iterations... 3.3x more than prior state-of-the-art... More importantly, it was the only method to reach diamond-tier at all."
**Context:** Compared against ReAct, Reflexion, and AutoGPT adapted for Minecraft.
**Confidence:** high

**Claim: Voyager uses code as its action space instead of low-level motor commands, representing temporally extended and compositional actions as executable programs.** [^1^]
**Source:** Voyager (Wang et al., 2023)
**URL:** https://arxiv.org/html/2305.16291
**Date:** 2023
**Excerpt:** "We opt to use code as the action space instead of low-level motor commands because programs can naturally represent temporally extended and compositional actions, which are essential for many long-horizon tasks in Minecraft."
**Context:** This is Voyager's key architectural decision - it generates JavaScript/Mineflayer code, not keyboard/mouse actions.
**Confidence:** high

**Claim: Voyager's exploration is driven entirely by the LLM planner (automatic curriculum). There is no learned exploration policy - 'where to go' is produced by the planner reasoning over the agent's current state and exploration history.** [^1^]
**Source:** Voyager (Wang et al., 2023)
**URL:** https://arxiv.org/html/2305.16291
**Date:** 2023
**Excerpt:** "The automatic curriculum takes into account the exploration progress and the agent's state to maximize exploration. The curriculum is generated by GPT-4 based on the overarching goal of 'discovering as many diverse things as possible'."
**Context:** The curriculum agent proposes the next task; the action agent generates code to execute it. There is no separate learned exploration module.
**Confidence:** high

### 7.2 GITM: Ghost in the Minecraft

**Claim: GITM uses a hierarchical LLM-based approach with three components: LLM Decomposer (breaks goals into sub-goals), LLM Planner (generates structured action sequences), and LLM Interface (maps structured actions to keyboard/mouse operations).** [^3^]
**Source:** Ghost in the Minecraft: Generally Capable Agents for Open-World Environments via Large Language Models with Text-based Knowledge and Memory (Zhu et al., arXiv 2023)
**URL:** https://arxiv.org/abs/2305.17144
**Date:** 2023-05-26
**Excerpt:** "The proposed LLM-based agent consists of an LLM Decomposer, an LLM Planner, and LLM Interface, which are responsible for the decomposition of sub-goals, structured actions, and keyboard/mouse operations, respectively."
**Context:** GITM's key innovation is using structured actions as an intermediate layer between LLM reasoning and low-level control.
**Confidence:** high

**Claim: GITM defines 9 structured actions: {equip, explore, approach, mine/attack, dig_down, go_up, build, craft/smelt, apply}. The explore action uses BFS on the surface and DFS underground.** [^3^]
**Source:** GITM (Zhu et al., 2023)
**URL:** https://arxiv.org/pdf/2305.17144
**Date:** 2023
**Excerpt:** "The set of structured actions is {equip, explore, approach, mine/attack, dig_down, go_up, build, craft/smelt, apply}... The explore action traverses the world until object is visible. This action regards the world as a chessboard, and each node on the chessboard is the center point of a 20x20 units area. Two strategies are implemented: when the agent is on the ground, the BFS explore will be adopted. When the agent is under the ground, mainly for exploring ore, the DFS explore will be adopted."
**Context:** The explore action is a SCRIPTED algorithm (BFS/DFS on a chessboard discretization), NOT a learned policy. The LLM planner decides WHEN to call "explore(object_name)" but the actual exploration behavior is hardcoded.
**Confidence:** high

**Claim: GITM achieves 67.5% success rate on ObtainDiamond, improving SOTA (VPT at 20%) by 47.5%. It unlocks 100% of the Minecraft Overworld Technology Tree (all 262 items), the first agent to do so.** [^4^]
**Source:** GITM GitHub results page
**URL:** https://github.com/OpenGVLab/GITM
**Date:** 2023
**Excerpt:** "GITM achieves the ObtainDiamond with 67.5% success rate, improving SOTA (VPT) by 47.5%... GITM is able to unlock the whole tech tree in Minecraft Overworld and obtain all items in it, which is the first to achieve."
**Context:** GITM requires no GPUs, just 2 days on a 32-core CPU node - 10,000x more efficient than VPT.
**Confidence:** high

**Claim: GITM uses text-based knowledge from Minecraft wikis/recipes and a text-based memory that stores successful plans. The LLM planner retrieves past plans from memory to inform future planning.** [^3^]
**Source:** GITM (Zhu et al., 2023)
**URL:** https://arxiv.org/pdf/2305.17144
**Date:** 2023
**Excerpt:** "Text-based Memory is designed for LLM to maintain common reference plans for each encountered objective as experimental knowledge... once the goal is achieved, the entirely executed action list would be stored in memory... To identify a common reference plan suitable for general situations, essential actions from multiple plans are summarized."
**Context:** Unlike Voyager's code-based skill library, GITM stores textual action lists and uses LLM summarization to distill common reference plans.
**Confidence:** high

**Claim: In GITM, exploration intelligence is SPLIT: the LLM planner decides WHAT to explore (e.g., 'explore diamond'), but the actual navigation algorithm (BFS/DFS chessboard traversal) is scripted, not learned.** [^3^]
**Source:** GITM (Zhu et al., 2023)
**URL:** https://arxiv.org/pdf/2305.17144
**Date:** 2023
**Excerpt:** "explore: The explore action traverses the world until object is visible... When the agent is on the ground, the BFS explore will be adopted... The action succeeds when the object is visible. The action fails when the explore exceeds a preset steps of 10,000."
**Context:** This is a critical finding for the decision fork. GITM exemplifies the "planner decides what, scripted algorithm does how" pattern for exploration.
**Confidence:** high

### 7.3 JARVIS-1: Multimodal Memory

**Claim: JARVIS-1 comprises a memory-augmented multimodal language model (MLM) that produces plans and a low-level action controller. It uses MineCLIP for visual understanding chained with an LLM for planning.** [^5^]
**Source:** JARVIS-1: Open-World Multi-task Agents with Memory-Augmented Multimodal Language Models (Wang et al., TPAMI 2025)
**URL:** https://arxiv.org/abs/2311.05997
**Date:** 2023-11-10
**Excerpt:** "JARVIS-1 comprises a memory-augmented multimodal language model (MLM) that produces plans and a low-level action controller. JARVIS-1 also utilizes a multimodal memory to store and obtain experiences as references for planning."
**Context:** JARVIS-1 chains MineCLIP (multimodal foundation model) with an LLM for multimodal planning.
**Confidence:** high

**Claim: JARVIS-1's multimodal memory is a key-value memory where keys are multimodal (task + observation/situation) and values are successfully executed plans. Retrieval uses multimodal queries combining task decomposition with visual observations.** [^5^]
**Source:** JARVIS-1 (Wang et al., 2023)
**URL:** https://arxiv.org/html/2311.05997
**Date:** 2023
**Excerpt:** "From a high level, it is a key-value memory where the keys are multimodal, comprising both the task and the observation (or situation) made when this memory entry was created. The values are the plans that were successfully executed."
**Context:** Unlike Voyager's code skill library or GITM's text memory, JARVIS-1 stores multimodal experience tuples (task, observation) -> plan.
**Confidence:** high

**Claim: JARVIS-1 achieves 12.5% success rate on ObtainDiamondPickaxe, a 5x improvement over prior SOTA. It can complete over 200 different tasks.** [^5^]
**Source:** JARVIS-1 (Wang et al., 2023)
**URL:** https://arxiv.org/abs/2311.05997
**Date:** 2023
**Excerpt:** "JARVIS-1 has achieved a completion rate of 12.5% in the long-horizon diamond pickaxe task. This represents a significant increase up to 5 times compared to previous records."
**Context:** Uses control and observation space similar to humans.
**Confidence:** high

**Claim: JARVIS-1 self-improves through a 'self-instruct' mechanism where it generates its own exploration curriculum, executes tasks, stores experiences in memory, and progressively enhances planning quality over time.** [^5^]
**Source:** JARVIS-1 (Wang et al., 2023)
**URL:** https://arxiv.org/html/2311.05997
**Date:** 2023
**Excerpt:** "JARVIS-1 can strengthen its own planning skills through exploration with its own proposed tasks (self-instruct) and a growing memory that helps with better planning on tasks that has been (partially) visited before... the curriculum almost follows the technical tree-growing direction."
**Context:** Similar to Voyager's automatic curriculum but with multimodal memory integration.
**Confidence:** high

**Claim: JARVIS-1 dispatches plans to a low-level goal-conditioned controller (not code generation like Voyager). The planner produces sub-goals; the controller executes them.** [^5^]
**Source:** JARVIS-1 (Wang et al., 2023)
**URL:** https://arxiv.org/html/2311.05997
**Date:** 2023
**Excerpt:** "The plans will be ultimately dispatched to the goal-conditioned controllers."
**Context:** This is a key difference from Voyager: JARVIS-1 uses a learned controller for low-level execution, while Voyager generates code that directly interacts with the environment API.
**Confidence:** high

### 7.4 Plan4MC: LLM Planner + RL Policy

**Claim: Plan4MC converts multi-task learning into learning basic skills (Finding-skills, Manipulation-skills, Crafting-skills) and planning over them. It uses RL with intrinsic rewards to acquire skills and an LLM-assisted skill graph for planning.** [^6^]
**Source:** Skill Reinforcement Learning and Planning for Open-World Long-Horizon Tasks (Yuan et al., ICLR 2023 / arXiv 2023)
**URL:** https://arxiv.org/html/2303.16563v2
**Date:** 2023-03-29
**Excerpt:** "We convert the multi-task learning problem into learning basic skills and planning over the skills... We propose three types of fine-grained basic skills in Minecraft, and use RL with intrinsic rewards to acquire skills."
**Context:** Plan4MC is the clearest example of the LLM+RL hybrid architecture.
**Confidence:** high

**Claim: Plan4MC's Finding-skill uses a hierarchical RL policy: a high-level LSTM policy trained with PPO to select exploration directions (maximizing unique grid cells visited) and a low-level DQN policy for goal-reaching navigation.** [^6^]
**Source:** Plan4MC (Yuan et al., 2023)
**URL:** https://arxiv.org/html/2303.16563v2
**Date:** 2023
**Excerpt:** "For the high-level policy of Finding-skills, which observes the agent's past locations, we use an LSTM policy and train it with truncated BPTT... The high-level policy is trained using PPO to maximize the number of unique grid cells visited."
**Context:** This is a LEARNED exploration policy, not scripted. The LSTM takes (x,y) coordinates and outputs directional goals.
**Confidence:** high

**Claim: Plan4MC uses the LLM ONLY for constructing a skill graph in advance (finding relationships between skills). During online execution, a skill search algorithm walks the pre-generated graph - the LLM is NOT involved in real-time planning.** [^6^]
**Source:** Plan4MC (Yuan et al., 2023)
**URL:** https://arxiv.org/html/2303.16563v2
**Date:** 2023
**Excerpt:** "Unlike previous LLM-based planning methods, we propose the skill graph and the skill search algorithm for interactive planning. The LLM only assists in the generation of the skill graph before task execution, avoiding uncontrollable failures caused by the LLM."
**Context:** This is a crucial architectural decision. Plan4MC deliberately removes the LLM from the online planning loop to avoid LLM latency and unreliability.
**Confidence:** high

**Claim: Plan4MC requires only 7M environmental steps total to train all skills and can unlock the iron pickaxe in the Minecraft Tech Tree. This greatly outperforms all other demonstration-free RL methods.** [^6^]
**Source:** Plan4MC (Yuan et al., 2023)
**URL:** https://arxiv.org/html/2303.16563v2
**Date:** 2023
**Excerpt:** "Plan4MC totally takes 7M environmental steps in training, and can unlock the iron pickaxe in the Minecraft Tech Tree in test. The sample efficiency greatly outperforms all other existing demonstration-free RL methods."
**Context:** A model-based RL method takes over 10M steps just to harvest cobblestones.
**Confidence:** high

**Claim: In Plan4MC, exploration intelligence lives in the RL-trained Finding-skill (learned LSTM policy), NOT in the LLM planner. The LLM only handles static skill graph construction offline.** [^6^]
**Source:** Plan4MC (Yuan et al., 2023)
**URL:** https://arxiv.org/html/2303.16563v2
**Date:** 2023
**Excerpt:** "The Finding-skill in Plan4MC can be replaced with any learning-to-explore RL policy... We use RL with intrinsic rewards to acquire skills... The LLM only assists in the generation of the skill graph before task execution."
**Context:** This is the strongest evidence that learned exploration policies (RL) can be effective, and that separating LLM planning from learned control is viable.
**Confidence:** high

### 7.5 DEPS, RL-GPT, STEVE-2/HAS, Optimus-1/2, Odyssey

#### DEPS (Describe, Explain, Plan and Select)

**Claim: DEPS is an interactive planning approach where a descriptor summarizes the current situation on failure, an explainer LLM locates errors in the previous plan, and a planner refines the plan. A learned goal selector ranks sub-goals based on estimated completion steps.** [^7^]
**Source:** Describe, Explain, Plan and Select: Interactive Planning with Large Language Models Enables Open-World Multi-Task Agents (Wang et al., NeurIPS 2023)
**URL:** https://arxiv.org/abs/2302.01560
**Date:** 2023-02-03
**Excerpt:** "DEPS facilitates better error correction on initial LLM-generated plan by integrating description of the plan execution process and providing self-explanation of feedback when encountering failures... it includes a goal selector, which is a trainable module that ranks parallel candidate sub-goals based on the estimated steps of completion."
**Context:** DEPS was the first zero-shot multi-task agent to accomplish 70+ Minecraft tasks.
**Confidence:** high

**Claim: DEPS with MrSteve (low-level controller with episodic memory) shows comparable or better performance than DEPS with Steve-1. The bed task (kill 3 sheep) shows huge improvement because MrSteve remembers where sheep are located.** [^8^]
**Source:** MrSteve: Instruction-Following Agents in Minecraft with What-Where-When Memory (Park et al., ICLR 2025)
**URL:** https://arxiv.org/abs/2411.06736
**Date:** 2024-11-11
**Excerpt:** "DEPS with MrSteve shows comparable performance or outperforms compared to DEPS with Steve-1. Especially, there is a huge performance gap between the two low-level controllers in the bed task, which requires killing three sheeps... MrSteve avoids this redundant exploration utilizing the episodic memory."
**Context:** This demonstrates that memory at the low-level controller level is critical for exploration efficiency.
**Confidence:** high

#### RL-GPT

**Claim: RL-GPT introduces a two-level hierarchical framework with a 'slow agent' that analyzes which actions are suitable for coding vs. RL, and a 'fast agent' that executes coding tasks. Coded actions with explicit conditions are executed sequentially; others are learned via RL.** [^9^]
**Source:** RL-GPT: Integrating Reinforcement Learning and Code-as-policy (Liu et al., NeurIPS 2024)
**URL:** https://arxiv.org/html/2402.19299v1
**Date:** 2024-02-29
**Excerpt:** "We introduce RL-GPT, a framework designed to enhance LLMs with trainable modules for learning interaction tasks... The slow agent analyzes actions suitable for coding, while the fast agent executes coding tasks... Actions falling outside this realm will be learned through RL."
**Context:** RL-GPT achieves SOTA on all designated MineDojo tasks and obtains diamonds within a single day on an RTX3090.
**Confidence:** high

**Claim: RL-GPT's key design decision is: 'Which actions should be learned with RL?' - high-level planning is amenable to direct coding, while low-level actions often necessitate task-specific refinement via RL.** [^9^]
**Source:** RL-GPT (Liu et al., 2024)
**URL:** https://arxiv.org/html/2402.19299v1
**Date:** 2024
**Excerpt:** "In embodied tasks, high-level planning is amenable to direct coding, while low-level actions often necessitate task-specific refinement, such as Reinforcement Learning (RL)."
**Context:** This directly answers the "where does intelligence live" question for RL-GPT: planning at LLM level, control at RL level.
**Confidence:** high

### 7.6 Exploration Strategy Comparison

#### How Each System Handles Exploration

| System | Exploration "What" | Exploration "How" | Where Intelligence Lives |
|--------|-------------------|-------------------|-------------------------|
| **Voyager** | LLM curriculum proposes next task/item to discover | Generated JavaScript code (Mineflayer API calls) | **Planner** - LLM decides what to explore; code is generated on-the-fly |
| **GITM** | LLM Decomposer identifies target objects | Scripted BFS (surface) / DFS (underground) on chessboard grid | **Split** - Planner decides WHAT to find; hardcoded algorithm does HOW |
| **JARVIS-1** | MLM planner generates sub-goal sequences | Goal-conditioned low-level controller (learned policy) | **Planner** - MLM decides sub-goals; controller executes navigation |
| **Plan4MC** | Skill graph search algorithm (NOT LLM at runtime) | RL-trained Finding-skill (LSTM high-level + DQN low-level) | **Learned Policy** - RL handles exploration; LLM only offline |
| **DEPS** | LLM planner generates/refines plans on failure | Steve-1 or MrSteve low-level controller | **Planner** with error correction; controller has episodic memory |
| **RL-GPT** | Slow LLM agent decides what to code vs. learn via RL | RL for actions not suitable for coding | **Split** - LLM for plan; RL for fine-grained control |
| **Odyssey** | MineMA fine-tuned LLM for skill selection | Open-world skill library (183 compositional skills) | **Planner** - Fine-tuned LLM selects from skill library |

**Claim: MrSteve's count-based exploration with VPT-Nav significantly outperforms Plan4MC's LSTM exploration policy on both map coverage and revisit count metrics.** [^8^]
**Source:** MrSteve (Park et al., ICLR 2025)
**URL:** https://arxiv.org/html/2411.06736v3
**Date:** 2024
**Excerpt:** "the Count-Based goal selector that directs the agent to the least-visited locations as goals and the VPT-Nav that effectively reaches those goals resulted in strong exploratory behavior... Plan4MC's high-level RNN policy struggled to keep track of visited places as the trajectory got longer, leading to a high Revisit Count."
**Context:** Count-based methods can outperform learned LSTM policies for pure exploration.
**Confidence:** high

**Claim: Task-conditioned exploration (using text prompts about likely locations for target resources) explores task-relevant locations more quickly than task-free exploration in early stages.** [^8^]
**Source:** MrSteve (Park et al., ICLR 2025)
**URL:** https://arxiv.org/html/2411.06736v3
**Date:** 2024
**Excerpt:** "the task-conditioned exploration method explores task-relevant locations more quickly in the early stages... For example, the text prompt for log is set to 'near pasture', while the text prompt for sand is set to 'near desert.'"
**Context:** Biasing exploration with LLM world knowledge about biome-entity relationships improves efficiency.
**Confidence:** high

### 7.7 Consensus Pattern for "Find Distant Resource"

Based on the review of 10+ major LLM-as-planner architectures for Minecraft, the following consensus patterns emerge for the specific question of **"find a distant resource"** (e.g., diamond ore, specific mob):

#### 1. The Universal Pattern: LLM Planner + Low-Level Controller

**Claim: All successful LLM-as-planner systems for Minecraft adopt a hierarchical two-level architecture: an LLM/MLM-based high-level planner that reasons about WHAT to do, and a low-level controller (code generator, learned policy, or scripted action) that handles HOW to execute.** [^1^] [^3^] [^5^] [^6^] [^7^] [^9^] [^12^]
**Source:** Multiple papers (Voyager, GITM, JARVIS-1, Plan4MC, DEPS, RL-GPT, Optimus-2)
**URL:** Multiple
**Date:** 2023-2025
**Excerpt:** (from RL-GPT) "high-level planning is amenable to direct coding, while low-level actions often necessitate task-specific refinement, such as Reinforcement Learning"
**Context:** This is the dominant architectural pattern across the entire field. No successful system maps LLM directly to keyboard/mouse actions.
**Confidence:** high

#### 2. Exploration Intelligence: Split Between Planner and Controller

**Claim: For the specific task of finding distant resources, exploration intelligence is SPLIT between the planner and the controller across all systems reviewed:**

- **The planner (LLM/MLM) handles:** WHAT resource to find, WHERE it might be located (biome knowledge), WHAT tools are needed, and WHEN to switch from exploration to exploitation
- **The controller handles:** The actual navigation, obstacle avoidance, terrain traversal, and low-level movement patterns

**Source:** Cross-paper analysis
**URL:** Multiple
**Date:** 2023-2025
**Confidence:** high

#### 3. Three Distinct Approaches to "How to Explore"

The systems reviewed fall into three categories for low-level exploration:

**Category A: Code Generation (Voyager)**
- The LLM generates JavaScript/Mineflayer code that calls environment APIs for navigation
- Exploration behavior emerges from code the LLM writes (e.g., `exploreCave()`, `findDiamond()`)
- Most flexible but most dependent on LLM quality

**Category B: Scripted Structured Actions (GITM)**
- The LLM decides WHEN to call `explore(object)` but the exploration itself is a scripted BFS/DFS algorithm
- Most reliable but least adaptive to complex terrain
- Uses A* for pathfinding after object is visible

**Category C: Learned RL Policy (Plan4MC, RL-GPT)**
- A trained RL policy handles exploration (LSTM for direction selection + DQN for goal-reaching)
- Most sample-efficient for repeated exploration tasks
- Generalizes to unseen environments better than scripts

#### 4. Memory is Critical for Efficient Exploration

**Claim: All high-performing systems incorporate some form of memory for exploration: Voyager's skill library, GITM's text-based plan memory, JARVIS-1's multimodal experience memory, MrSteve's Place Event Memory, and Optimus-1's hybrid memory.** [^1^] [^3^] [^5^] [^8^] [^11^]
**Source:** Cross-paper analysis
**URL:** Multiple
**Date:** 2023-2025
**Excerpt:** (from JARVIS-1) "Multimodal memory facilitates planning using both pre-trained knowledge and its actual game survival experiences"
**Context:** Memory prevents re-exploring already-visited areas and enables return to previously discovered resources.
**Confidence:** high

#### 5. Key Recommendation for AI Utopia

For a multi-agent Minecraft village with Ray RLlib PPO + LSTM for low-level control and an LLM planner at the top:

**The consensus strongly supports the existing architecture (LLM planner + RL low-level controller).** The LLM planner should:
1. **Decide WHAT to explore** (which resource, which direction, which biome)
2. **Provide the exploration goal** to the RL low-level controller as a target (e.g., "find forest biome", " locate iron ore")
3. **NOT** generate low-level movement commands directly

The RL low-level controller (PPO + LSTM) should:
1. **Handle the actual navigation** to the target
2. **Learn terrain-specific movement patterns** through RL
3. **Maintain episodic memory** of visited locations (like MrSteve's PEM)

The "Finding-skill" pattern from Plan4MC is the most directly applicable: train a specialized RL policy for exploration that the LLM planner can invoke as a primitive skill.
