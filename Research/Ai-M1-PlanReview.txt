## Must-fix (would break smoke or Plan B)

### 1. PettingZoo agent id vs Carpet player name

Today:
- Env agents are `gatherer_0`, `builder_0`, etc. (`wrapper.py`)
- Carpet spawn uses pool names like `Bjorn` (`agent.py`)
- M1A `observationsAll()` keys JSON by **player name**

So `env.step({"gatherer_0": act})` will dispatch to a non-existent player, while obs come back keyed as `"Bjorn"`. M0 hid this because Java returned `{}`.

**Fix direction (pick one and document it in the plan):**
- **A)** Use Carpet `agent_name` as the PettingZoo agent id everywhere, or  
- **B)** Keep `gatherer_0` but add an explicit `agent_id → player_name` map (identity DB + env config), used in `_read_all_obs`, `dispatch_skill`, and smoke tests.

Task 19 (role plumbing) does not solve this.

### 2. Missing Python obs decode layer (plan mentions it, no task implements it)

`CoreObsBuilder` emits `agent_uuid`, `role_id`, and **string** `minecraft:oak_log` item ids. Gym spaces expect:
- `agent_uuid_embed` (384-d)
- `role_one_hot` (4,)
- `inv_slot_item_ids` as `MultiDiscrete(1024)`

Task 15 only patches `goal_embedding`. There is no task for:
- UUID → embed (even a deterministic hash embed is fine for M1A)
- `role_id` → one-hot
- item string → integer id (stable hash into `[0, 1024)` is enough for pipeline)
- `goal_ticks_left` (can decrement in Python from stub subgoal)

Without this, `_decode_obs()` will produce wrong dtypes/shapes and reward tests that use string inventory keys won’t match live obs.

### 3. Action mask inputs not emitted by Java

`compute_gatherer_action_mask()` expects `target_chest_in_range` and `target_resource_in_range`. `CoreObsBuilder` / `GathererOverlayBuilder` never set them. Masks will be wrong (e.g. HARVEST always allowed or always blocked depending on defaults).

Add small booleans in Java (or derive in Python from `g_nearest_resources` / chest scan) and document in T9/T10.

### 4. Episodic memory not wired in T16 (contradicts plan goal)

The header says M1A replaces “count-only stub” with real Chroma writes in the loop. T17 implements the writer, but **T16 never calls** `importance_score()` / `EpisodicMemoryWriter.maybe_write()` in `step()`.

The completion checklist (“`memory inspect` returns records after a few drives”) only works if you either:
- Wire memory in `env.step()` (spec §4.6 step 8), or  
- Add a separate task that `drive` + manual write path uses.

Right now T17 alone does not satisfy the stated goal.

### 5. `advance_tick_await_events` semantics vs 20 Hz RL (Plan B landmine)

`Py4JEntryPoint` docstring says “advance one tick”; M1A `MotorBridge` only **polls a completion queue** while the dedicated server ticks on its own.

For `agent drive` + live server: **OK** (server runs at 20 TPS; blocking until completion works).

For `env.step()` at 20 Hz in Plan B: **problematic** if each step blocks until skill completion. A NAVIGATE can take hundreds of ticks; one `step()` would wall-clock for seconds and preempt logic would abort skills every tick if you dispatch every step.

**Recommendation for M1A:** Document explicitly: “M1A `env.step()` may block until completion or timeout; Plan B must change to **one RL step ≈ one server tick**, skills persist across steps, completions are non-blocking.” Optionally add a stub `advanceOneTick()` on Java in M1A so the API exists.

Also: preempt-on-new-dispatch is fine only if the policy mostly emits WAIT while a skill runs — worth stating in Plan B training notes.

### 🚨 Critical Fix (Must Fix before execution)

**1. The "Phasing Through Walls" Bug (`NavigateSkill`, `HarvestSkill`, `DepositChestSkill`)**
In Tasks 2, 3, and 4, the movement logic calculates a delta and moves the agent using:
```java
agent.setPosition(next.x, next.y, next.z);
```
**Why this fails:** `setPosition()` literally teleports the entity, bypassing Minecraft's collision detection engine and gravity. The agent won't "press against" a tree or hill as the plan describes; it will phase directly through solid terrain, and if the target is in the air, it will fly. For an RL agent, this ruins spatial learning because the agent will learn to clip through walls instead of navigating around them.
**The Fix:** You have two options here:
*   *Option A (Programmatic):* Use `agent.move(net.minecraft.entity.MovementType.SELF, stepVec)`. This runs the movement through vanilla AABB collision logic, so the agent will bump into walls and fall with gravity.
*   *Option B (Carpet ActionPack - Recommended):* Since Carpet is already installed, fetch the agent's `EntityPlayerActionPack` and send `forward` and `look` synthetic inputs. This is much safer as it natively handles gravity, collision, and even 1-block auto-stepping without you having to simulate physics.

### 🟡 Should-Fixes (High confidence improvements)

**2. Python `hash()` Non-Determinism (Task 13 - `exploit.py`)**
In `_inv_hash`, the inventory is hashed using Python's built-in `hash(tuple(...))`. 
**Why this fails:** Python's `hash()` uses SIPHash with a random seed generated at process startup. While it's consistent *during* a run, it will produce different values across different runs. The spec demands a strict determinism harness (comparing weights and actions across runs); random hashes in logs and replay traces will create flaky determinism diffs.
**The Fix:** Since `INV_REPEAT_BUF_SIZE` is only 10, don't hash it at all. Python tuples are directly comparable. Just store the raw `tuple(zip(ids, counts))` directly in the deque and use `list(self.inv_snapshots).count(snap)`.

**3. Java `Math.abs` Edge Case (Task 10 - `GathererOverlayBuilder`)**
```java
return Math.abs(Registries.ENTITY_TYPE.getId(e.getType()).toString().hashCode() % 256) / 256.0;
```
Because of how two's complement math works, `Math.abs(Integer.MIN_VALUE)` is a negative number. While the `% 256` inside the `abs()` saves this from crashing, it's a known Java gotcha that's better avoided.
**The Fix:** Use a bitwise AND to mask the sign bit instead: 
```java
return (id.hashCode() & 0xFF) / 256.0;
```

**4. `GathererOverlayBuilder` Y-Level Scan (Task 10)**
The resource grid scan loop goes from `dy = 3` down to `-3` and `break`s on the first non-air block. This effectively generates a top-down surface heightmap. 
This is perfectly fine for the M1 Gatherer (who only needs surface wood/crops), but won't "see" ores if they are exposed in the wall of a cave underneath a surface block. Just verifying this is the intended M1-Pipeline simplification!

### ❓ Questions for Alignment

1. **Elevation / Step-height:** If we switch to `agent.move()` to fix the collision bug, the agent will get stuck on 1-block elevation changes (like a hill) unless it jumps. Do we want to implement jumping logic in `NavigateSkill`, or just accept that the agent will get stuck on hills during M1? (This is why the Carpet ActionPack is great—it can handle standard walk physics).
2. **Scan Radius Performance:** `HarvestSkill` scans a 17x17x17 cube (~4,900 blocks) synchronously on the main server thread every time it needs a target. For M1 with 4 agents, this is totally fine, but it might create micro-stutter if heavily spammed at 20 TPS. Are we okay leaving this O(R³) scan as-is for M1-Pipeline and optimizing it with an event-driven cache in M2?
